# Context Maintenance Rules

This document outlines the rules and procedures for maintaining consistent context across the Neothink Network State Ecosystem development.

## Rule 1: Priority Hierarchy

When making decisions or implementing features, consider this priority order:

1. **Immediate Implementation**
   - Core membership platform functionality
   - Essential user features
   - Critical business logic

2. **Near-Term Features**
   - Platform-specific components
   - Basic integration points
   - Community features

3. **Medium-Term Expansion**
   - Cross-platform functionality
   - Advanced features
   - Enhanced integrations

4. **Long-Term Vision**
   - Token economics
   - Governance systems
   - Network State evolution

## Rule 2: Documentation Standards

### Decision Documentation Format
```markdown
DECISION [YYYYMMDD]: [Brief Title]
SCOPE: [Platform(s) affected]
DETAILS: [Concise explanation]
RATIONALE: [Key reasons]
IMPLICATIONS: [Effects on other components/timeline]
```

### Implementation Documentation Requirements
- Technical specifications
- User experience flows
- Data models
- API endpoints
- Integration points
- Security considerations

### Context Updates
- Document all context-changing decisions
- Update relevant platform documentation
- Notify team members of significant changes
- Maintain version history

## Rule 3: Question Classification

When addressing implementation questions, classify them as:

1. **Technical Implementation**
   - How features will be built
   - Technical architecture decisions
   - Integration methods

2. **User Experience**
   - Member interactions
   - Interface design
   - Progress tracking
   - Community engagement

3. **Content Structure**
   - Course organization
   - Resource management
   - Documentation hierarchy
   - Knowledge base structure

4. **Business Logic**
   - Subscription handling
   - Access control
   - Progress requirements
   - Cross-platform rules

5. **Vision Alignment**
   - Long-term goals
   - Network State progression
   - Governance evolution
   - Token integration

## Rule 4: Knowledge Gap Protocol

When encountering missing information:

1. **Identify the Gap**
   - What information is missing
   - Why it's needed
   - Impact on implementation

2. **Document Dependencies**
   - What can't proceed without this information
   - What can proceed in parallel
   - Temporary workarounds

3. **Resolution Path**
   - Who can provide the information
   - Alternative approaches
   - Timeline implications

## Rule 5: Cross-Platform Considerations

When implementing features:

1. **Check Platform Impact**
   - How it affects each platform
   - Integration requirements
   - Shared resources

2. **Maintain Consistency**
   - User experience
   - Terminology
   - Technical patterns
   - Business rules

3. **Future-Proof Design**
   - Token integration points
   - Governance hooks
   - Network State evolution
   - Physical/digital fusion

## Rule 6: Guardian Considerations

When implementing features that will eventually be community-controlled:

1. **Document Current Control**
   - What the guardians control
   - How decisions are made
   - Current limitations

2. **Plan for Transition**
   - What triggers transition
   - How control transfers
   - Technical requirements

3. **Build for Both Phases**
   - Guardian-controlled phase
   - Community-controlled phase
   - Transition mechanisms

## Rule 7: Regular Reviews

Conduct regular documentation and context reviews:

1. **Weekly**
   - Update implementation status
   - Document new decisions
   - Review knowledge gaps

2. **Monthly**
   - Review documentation accuracy
   - Update priority alignments
   - Adjust timelines

3. **Quarterly**
   - Evaluate vision alignment
   - Update long-term plans
   - Revise documentation structure

## Maintaining These Rules

These rules should be:
1. Reviewed quarterly
2. Updated as needed
3. Approved by guardians
4. Communicated to all team members
5. Enforced consistently
